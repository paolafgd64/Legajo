====================================================
1) ESTRUCTURA MENTAL DEL PROYECTO
====================================================

- Legajo/Legajo/urls.py
	Mapa de rutas (pantallas y APIs). Si preguntan "por donde entra esta funcionalidad", empieza aqui.

- Legajo/web/views.py
	Capa HTTP: recibe request, valida sesion/rol, llama servicios, devuelve HTML/JSON/redirect.

- Legajo/web/services/
	Logica de negocio reutilizable.
	- books.py: libros
	- users.py: importacion de usuarios
	- exchanges.py: solicitud de intercambio
	- cloudinary.py: integracion externa de imagenes

- Legajo/web/validators/
	Reglas de validacion y errores controlados.

- Legajo/web/templates/web/
	Pantallas HTML.

Regla simple para explicar al jurado:
- views = entrada/salida web
- services = reglas del negocio
- validators = calidad de datos
- templates = interfaz

====================================================
7) SERVICES Y VALIDATORS (EXPLICACION PROFUNDA)
====================================================

7.1 Que problema resuelve cada capa

- Service (web/services):
	- Recibe datos ya parseados por la view.
	- Aplica reglas de negocio del dominio (quien puede, cuando, bajo que estado).
	- Orquesta modelos y transacciones.
	- Devuelve datos listos para responder (dict/list) o lanza error controlado.

- Validator (web/validators):
	- Revisa formato y consistencia minima de entrada.
	- Normaliza texto/tipos para evitar basura en DB.
	- Lanza ValidationServiceError con mensaje claro para frontend.

Resumen corto para jurado:
- validator = "esto tiene forma valida"
- service = "esto tiene sentido de negocio"

7.2 Contrato entre view -> service -> validator

1) La view recibe HTTP y parsea (JSON/POST/FILES).
2) La view llama un service.
3) El service llama validator(es) para limpiar/validar payload.
4) Si todo esta bien, el service persiste en DB.
5) Si hay error, se levanta ControlledError y la view responde JSON con status correcto.

Por que esto es bueno:
- Evita duplicar validaciones en varias views.
- Permite reutilizar negocio desde endpoints distintos.
- Facilita testing unitario por capa.

7.3 Services del proyecto (que hace cada uno)

A) web/services/books.py
- list_books(user, filters):
	- Construye consulta base de libros activos.
	- Si no es admin, restringe a libros del propietario autenticado.
	- Aplica filtros (q, titulo, autor, usuario, genero, estado).
	- Retorna lista serializada para frontend.

- create_book(user, data, image_file=None):
	- Valida payload con validate_book_payload.
	- Resuelve URL de imagen (Cloudinary o local).
	- Crea libro, autor y genero dentro de transaction.atomic.
	- Devuelve libro serializado.

- update_book(user, libro_id, data):
	- Verifica permiso con _get_book_for_user.
	- Revalida campos y actualiza entidades relacionadas.
	- Mantiene consistencia autor/genero usando set().

- soft_delete_book(user, libro_id):
	- No borra fisico: marca activo=False (borrado logico).

- import_books_from_payload(payload):
	- Carga masiva con control de duplicados por (usuario, titulo).
	- Cuenta creados/omitidos para feedback administrativo.

B) web/services/users.py
- _normalize_user_payload:
	- Convierte aliases de campos del JSON a formato interno.
	- Verifica requeridos, telefono numerico y rol valido.

- import_users_from_payload(payload, actualizar=False):
	- Recorre lista y crea o actualiza usuarios segun bandera.
	- Maneja password con set_password/create_user (hash seguro).
	- Retorna resumen: creados, actualizados, omitidos.

C) web/services/exchanges.py
- request_exchange(user, data):
	- Valida libroId.
	- Verifica que el libro exista y no sea del mismo usuario.
	- Evita duplicar solicitudes pendientes al mismo libro.
	- Crea intercambio en estado PENDIENTE.

7.4 Validators del proyecto (que protege cada uno)

A) web/validators/books.py
- validate_required_fields:
	- Requeridos obligatorios cuando aplica.
- validate_book_payload:
	- Normaliza campos y estado por defecto.
	- Controla longitud maxima de titulo/genero/url.
	- Entrega payload limpio para persistencia.

B) web/validators/exchanges.py
- validate_exchange_payload:
	- Exige libroId y lo convierte a entero.
	- Si falla, lanza ValidationServiceError explicito.

C) web/validators/errors.py
- Jerarquia de errores controlados:
	- ValidationServiceError -> 400
	- PermissionDeniedServiceError -> 403
	- NotFoundServiceError -> 404
	- DatabaseServiceError -> 500
	- ExternalServiceError -> 502

Idea clave para defender:
"No devolvemos trazas tecnicas al cliente; devolvemos errores controlados, entendibles y con codigo HTTP correcto."

7.5 Ejemplo completo (registro de libro)

Flujo real:
1) view api_libros (POST) recibe request.POST + request.FILES.
2) llama create_book(...) en service.
3) create_book llama validate_book_payload(...).
4) validator limpia y valida reglas basicas.
5) service guarda libro/autor/genero en transaccion.
6) devuelve serialize_book(...).
7) view responde 201 JSON.

Si hay error:
- validator/service lanza ControlledError
- view responde mensaje + status correspondiente

7.6 Señales de buen diseno que puedes mencionar

- Existe desacople real entre HTTP y negocio.
- Se usan transacciones en operaciones criticas.
- Se aplica borrado logico para trazabilidad.
- La validacion no esta mezclada con templates.
- Hay manejo uniforme de errores de servicio.

8) GUIA EXPRESS PARA EXPLICAR EN 1 MINUTO
====================================================

"En este proyecto, las views no contienen toda la logica: solo reciben la peticion y responden. Las reglas del sistema viven en services, por ejemplo books.py para CRUD y carga masiva, users.py para importacion y exchanges.py para solicitudes de intercambio. Antes de guardar, los validators limpian y validan los datos. Si algo falla, se usan errores controlados con codigo HTTP correcto. Asi mantenemos el codigo mas mantenible, testeable y seguro."
